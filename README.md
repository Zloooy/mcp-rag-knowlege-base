# MCP RAG Knowledge Base

MCP-сервер, который превращает локальную папку с документами в поисковую базу знаний с поддержкой LLM на основе **Corrective RAG** (корректирующего цепного ответа). Построен на LangGraph, ChromaDB, гибридном поиске BM25 + векторы (RRF) и локальной модели через Ollama.

## Возможности

- **4 MCP-инструмента**: индексация документов, ответы на вопросы по базе знаний, поиск релевантных фрагментов, проверка статуса индекса
- **Corrective RAG**: переформулирование запроса → гибридный поиск (BM25 + вектор → RRF) → оценка релевантности → генерация ответа с циклами повторных попыток
- **Поддержка форматов**: `.md`, `.txt`, `.py`, `.js`, `.ts`, `.json`, `.yaml`
- **Умная разбивка на чанки**: разбиение кода с учётом структуры дерева (tree-sitter), сохранение заголовков при работе с markdown
- **Docker Compose**: сервер + Ollama — одна команда для запуска всего стека

## Быстрый старт

### Требования

- Docker & Docker Compose
- Ollama с установленной моделью (по умолчанию `qwen2.5:3b`)
- `uv` (менеджер пакетов Python) — для локальной разработки

### Запуск через Docker Compose

```bash
docker compose up
```

Сервер запустится на порту 8000 с HTTP-транспортом по адресу `http://localhost:8000/mcp`.

Докер автоматически смонтирует демо-документы из `./rag_dataset/demo_documents` внутрь контейнера. База данных ChromaDB будет сохранена в именованном томе `chroma_data`.

Настройки окружения можно переопределить через файл `.env`:

```bash
cp .env.example .env
# отредактируйте .env по необходимости
docker compose up
```

Ключевые переменные:

- `RAG_OLLAMA_MODEL` — имя модели для генерации текста (по умолчанию: `qwen2.5:3b`)
- `RAG_OPENAI_COMPLETIONS_BASE_URL` / `MODEL` / `API_KEY` — OpenAI-совместимый API для генерации ответов
- `RAG_OPENAI_EMBEDDINGS_BASE_URL` / `MODEL` / `API_KEY` — внешний эмбеддинг-сервис (например `nomic-embed-text`)
- `RAG_CHROMA_PERSIST_DIR` — директория хранения ChromaDB
- `RAG_CHUNK_SIZE`, `RAG_CHUNK_OVERLAP` — параметры разбиения на чанки

### Локальная разработка

```bash
# Установка зависимостей
uv sync

# Запуск сервера (stdio-транспорт для интеграции с IDE)
uv run .

# Или с HTTP-транспортом
uv run . --transport streamable-http --port 8000
```

## Подключение к VS Code / GitHub Copilot

1. Убедитесь, что MCP-сервер запущен (`docker compose up` или `uv run .`)
2. Откройте рабочую область проекта в VS Code
3. Файл `.vscode/mcp.json` уже присутствует — он указывает на `http://localhost:8000/mcp`
4. Откройте палитру команд VS Code (**Ctrl+Shift+P**) и выберите **GitHub Copilot: Connect to MCP Server** (или используйте значок MCP в боковой панели)
5. 4 инструмента сервера станут доступны агенту автоматически

Конфигурация MCP-сервера в VS Code (`.vscode/mcp.json`):
Чтобы агент воспринимал сервер как базу знаний по вселенной SCP, лучше назвать сервер соответствующе.
```json
{
  "servers": {
    "scp-knowledge-base": {
      "type": "http",
      "url": "http://localhost:8000/mcp"
    }
  }
}
```

## MCP-инструменты

| Инструмент | Описание |
|------------|----------|
| `rag_index_folder` | Сканировать папку, разбить документы на чанки, создать эмбеддинги, сохранить в ChromaDB |
| `rag_ask_question` | Задать вопрос проиндексированной базе знаний — запускает полный RAG-пайплайн |
| `rag_find_relevant_docs` | Найти релевантные фрагменты документов без генерации ответа (гибридный BM25+вектор) |
| `rag_index_status` | Проверить статистику индекса: количество файлов, чанков, время последней индексации |

## Примеры использования с промптами

Ниже приведены примеры запросов, основанные на тестовых данных из `tests/test_retrieval.py`, которые обращаются к демо-документам из `rag_dataset/demo_documents/`.

### Верификационные факты

Это сгенерированная текстовая RPG в стиле SCP Foundation — «Containment Breach» (нарушение содержания).

**Запрос:** *«Какие роли доступны игроку в scp_rpg.py и чем они различаются?»*

Инструмент `rag_ask_question` найдёт в коде класс `Player` и его конструктор, где определены три роли: **Researcher** (исследователь), **Security** (охранник) и **MTF** (Mobile Task Force). Каждая роль имеет разные характеристики HP, рассудка (sanity), навыков боя, скрытности и знаний, а также стартовое снаряжение.

**Запрос:** *«Какой код нужен для лифта на поверхность в игре scp_rpg.py?»*

Ответ: код **4729**. Он хранится в предмете `elevator_code` (документ «Elevator Code Note»), найденном в комнате `cafeteria` (Столовая), и используется для вызова лифта на поверхность из комнаты `entrance` (Входная зона) через действие `do_elevator()`.

**Запрос:** *«Что известно о причинах containment breach в Site-19, упомятой в коде scp_rpg.py?»*

Согласно тексту в `scp_rpg.py`, нарушение содержания было спровоцировано ИИ **SCP-079** («Old AI»). В документе `document_breach` (отчёт о нарушении), который можно подобрать в офисе, упоминается, что некий д-р ██████ получил повышенные привилегии обработки для терминала SCP-079 в 03:47, что привело к каскадному нарушению содержания.

**Запрос:** *«Какой предмет в scp_rpg.py защищает от потери рассудка и как он работает?»*

Предмет **SCP-714** (нефритовое кольцо) защищает от потери рассудка (sanity). При подборе в maintenance tunnels оно снижает потерю рассудка в 3 раза (`amount = amount // 3`), а также даёт бонус +30 к sanity. Дополнительно защищает от некоторых эффектов SCP-сущностей.

**Запрос:** *«Какие SCP-объекты присутствуют как аномальные сущности в scp_rpg.py?»*

В игре присутствуют 6 SCP-объектов: **SCP-173** (Скульптура), **SCP-096** (Скромный парень), **SCP-049** (Чумной доктор), **SCP-682** (Неуничтожимая ящерица), **SCP-079** (Старый ИИ) и **SCP-939** (Со многими голосами). Каждый представлен в словаре `SCP_DATA` со своими характеристиками HP, урона и особыми механиками взаимодействия.
